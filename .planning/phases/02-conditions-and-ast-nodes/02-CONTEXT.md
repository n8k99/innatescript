<decisions>
## Implementation Decisions

### AST node catalog
- 20 node kinds, each represented as a keyword constant:
  - `:program` — top-level wrapper, children are all statements
  - `:bracket` — `[context[verb[args]]]` bracket expressions
  - `:agent` — `(name)` agent address
  - `:bundle` — `{name}` bundle/scope reference
  - `:reference` — `@name` direct/indirect reference
  - `:search` — `![expr]` search directive
  - `:fulfillment` — `expr || (agent){instruction}` fulfillment chain
  - `:emission` — `-> values` emission statement
  - `:decree` — `decree name [body]` declaration
  - `:wikilink` — `[[Title]]` document reference
  - `:combinator` — `+word` scope extension
  - `:lens` — `{key:value}` filter/grouping on result sets
  - `:kv-pair` — `key: value` inside brackets
  - `:modifier` — `/word` presentation directive
  - `:prose` — non-executable text line (passthrough)
  - `:heading` — `# text` markdown heading
  - `:string-lit` — `"quoted string"`
  - `:number-lit` — integer literal
  - `:bare-word` — unquoted identifier
  - `:emoji-slot` — `<emoji>` type annotation
- No separate wikilink-with-brackets node — `[[sylvia[command]]]` tokenizes as three bracket levels, not a wikilink. The disambiguation happens in the tokenizer (Phase 3), not the AST.

### Node structure
- Single universal `defstruct node` with four slots: `kind` (keyword), `value` (any), `children` (list of nodes), `props` (plist)
- One constructor: `make-node` with `&key kind value children props`
- This is deliberate — a universal node means `etypecase` dispatches on `(node-kind n)`, not on struct type. Adding a new node kind requires zero struct changes.
- `props` carries kind-specific data:
  - `:reference` nodes: `(:qualifiers ("generative hard prompt") :combinator "all")`
  - `:lens` nodes: `(:rhs "==" )` or `(:rhs "Seed")`
  - `:kv-pair` nodes: child is the value node
- Accessors: `node-kind`, `node-value`, `node-children`, `node-props`

### Result types
- `defstruct innate-result` with slots: `value` (any), `context` (keyword — :query, :scope, :render, :commission)
- `defstruct resistance` with slots: `message` (string), `source` (string — the expression that failed)
- Predicate: `resistance-p` (auto-generated by defstruct)

### Condition hierarchy
- Three condition types, following the architecture researcher's recommendation:
  - `innate-condition` — base condition, parent of all Innate conditions
  - `innate-parse-error` (subtype of `innate-condition` and `error`) — syntax errors with `line` and `col` slots. This IS an error — malformed syntax is unrecoverable.
  - `innate-resistance` (subtype of `innate-condition` and `condition`, NOT `error`) — resolution failures. Uses `signal`, not `error`. Fulfillment (`||`) catches these via `handler-case`.
- No `innate-fulfillment-signal` — fulfillment is handled by the evaluator catching `innate-resistance`, not by signaling a separate condition.
- No `innate-resolution-error` — resolution failure IS resistance. Only parse errors are true errors.

### Package exports
- `innate.types` exports: `make-node`, `node-kind`, `node-value`, `node-children`, `node-props`, all 20 `+node-*+` constants, `make-innate-result`, `innate-result-value`, `innate-result-context`, `make-resistance`, `resistance-p`, `resistance-message`, `resistance-source`
- `innate.conditions` exports: `innate-condition`, `innate-parse-error`, `innate-resistance`, `parse-error-line`, `parse-error-col`, `resistance-condition-message`, `resistance-condition-source`
- `packages.lisp` must be updated with these exports so downstream packages can `:import-from` them

- Whether to define `print-object` methods for pretty-printing nodes and conditions
- Test structure for round-tripping node construction

<specifics>
## Specific Ideas

- Node kind constants use the `+name+` convention: `+node-bracket+`, `+node-prose+`, etc. — CL convention for constants
- The `resistance` struct is a data type (return value), while `innate-resistance` is a condition (signalable). They have the same slots but different purposes. The struct is what the evaluator passes around; the condition is what gets signaled up the call stack.

</specifics>
