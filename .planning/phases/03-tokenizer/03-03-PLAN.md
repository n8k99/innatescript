Modify `src/parser/tokenizer.lisp` to add three features to the tokenizer:

**1. Wikilink disambiguation (TOK-16):**

Replace the Plan 02 stub for `#\[` handling in the main dispatch. New logic:

```
When current char is #\[:
  capture start-line/start-col
  advance past first [
  if current char is also #\[:
    ;; Double bracket — disambiguate
    call %disambiguate-double-bracket
    %disambiguate-double-bracket is a pure lookahead:
      - save current pos as scan-pos (pointing at second [)
      - scan-pos = pos + 1 (past the second [)
      - loop: if char at scan-pos is ]:
          result is :wikilink, inner text is chars between second [ and this ]
          advance pos past the closing ]]
          emit (make-token :type :wikilink :value inner-text :line sl :col sc)
          return
        if char at scan-pos is [:
          result is :nested-bracket
          emit two :lbracket tokens (first at sl/sc, second at sl/sc+1)
          advance pos past only the second [ (let normal dispatch handle the third [)
          return
        else: accumulate char, increment scan-pos
      - if scan-pos reaches end of source: signal innate-parse-error "Unterminated [["
  else:
    ;; Single bracket
    emit (make-token :type :lbracket :value nil :line sl :col sc)
```

CRITICAL: The disambiguation function must NOT mutate pos. It reads ahead, decides, then the caller advances pos to the correct position.

**2. Prose detection (TOK-17):**

Add a `line-start-p` boolean variable initialized to `T`. Track it:
- Set to `T` at the beginning of tokenization
- Set to `T` after consuming a newline
- Set to `NIL` after emitting any non-newline token

At the top of the main loop, when `line-start-p` is T:
1. Skip leading whitespace (spaces/tabs) without emitting
2. Check `(current)`:
   - If nil: break (end of input)
   - If `#\Newline`: handle as newline (see below), continue
   - If char is in executable sigil set `(member (current) '(#\[ #\( #\{ #\@ #\! #\# #\/ #\> #\-))`: set `line-start-p` to NIL, fall through to normal dispatch
   - If char satisfies `alpha-char-p` or is `#\_`: peek-read the full bare word. If it equals `"decree"`, set `line-start-p` to NIL, emit :decree token, continue. Otherwise this whole line is prose — read remaining chars to EOL (or end of source), prepend the already-read word, emit :prose token with the full line text as value. Consume the newline if present but do NOT emit it as :newline here (prose implicitly ends at newline). Set `line-start-p` to T.
   - If char is `#\"` or `digit-char-p`: this is ambiguous at line start. Per the locked CONTEXT.md rules, these don't appear in the executable sigil set, so a line starting with a digit or a quote that isn't inside brackets is prose. Read to EOL and emit :prose.
   - Otherwise (any other char not in sigil set): read to EOL, emit :prose.

**3. Newline emission with collapse (TOK-18 support):**

Add a `last-was-newline` boolean initialized to NIL.

When the current char is `#\Newline`:
- capture start-line/start-col
- advance (this increments line, resets col)
- if `last-was-newline` is T: do NOT emit (collapse consecutive newlines)
- else: emit `(make-token :type :newline :value nil :line sl :col sc)`, set `last-was-newline` to T
- set `line-start-p` to T
- continue

When any non-newline token is emitted: set `last-was-newline` to NIL.

**Note on `-` at line start:** The CONTEXT.md sigil set includes `-`. Lines starting with `-` enter normal dispatch where `-` triggers the `->` check. If next char is not `>`, the current spec says signal `innate-parse-error`. However, `burg_pipeline.dpn` contains lines like `- "<emoji> Seed"`. For the integration test to pass, treat `-` followed by non-`>` at line start as prose (read remaining line, prepend `-`, emit :prose). This is the pragmatic choice from Research Note 5. Add a comment: `; Spec gap: - without > treated as prose at line start (burg_pipeline.dpn compat)`.

**Note on `>` at line start:** Similarly, `>` is in the sigil set but has no corresponding token type. If `>` appears at line start and is not part of `->`, treat the line as prose. Add a comment documenting this.

Write tests first (RED), then implement (GREEN).
